3. Project Description

The Bug Tracking System is a role-based web application designed to streamline the bug reporting and resolution workflow in software development teams. The application addresses the real-world problem of managing software defects efficiently by providing a centralized platform where testers can report bugs, administrators can assign them to developers, and developers can track and update their status.

This system implements three distinct user roles:

  • Tester: Can create new bug reports and assign them to developers
  • Developer: Can view assigned bugs and update their status
  • Administrator: Can view all bugs, manage assignments, and oversee the entire workflow

The application solves common pain points in bug management such as lost bug reports, unclear ownership, and lack of visibility into bug status. By centralizing bug tracking and enforcing clear workflows, teams can improve communication, reduce response times, and ensure no bugs fall through the cracks.

4. Features List

Core Features

  1. User Authentication and Authorization
    • Secure login system
    • Role-based access control (Admin, Developer, Tester)
    • User session management via AuthContext
  2. Bug Submission (Tester Role)
    • Create new bug reports with title and description
    • Set priority levels: low, medium, high, critical
    • Set severity levels: minor, major, critical
    • Assign bugs to available developers
    • Form validation (max 120 chars for title, 2000 for description)
  3. Bug Assignment (Admin & Tester Roles)
    • View complete list of all bugs in the system
    • Assign or reassign bugs to available developers via dropdown
    • See current assignee for each bug
    • Filter bugs by status, priority, or assignee
  4. Bug Status Management (Developer Role)
    • View only bugs assigned to the logged-in developer
    • Update bug status through workflow stages:
    • Status options: in_progress, resolved, rejected, closed
  5. Dashboard Views
    • Role-specific dashboards with relevant information
    • Real-time data updates after actions
    • Sortable and filterable bug tables
    • Color-coded status tags for quick visual identification
  6. Data Display
    • Bug title, description, priority, and severity
    • Reporter (creator) information with display name
    • Assignee information
    • Current status with color-coded tags
    • Timestamps for creation and last update

User Experience Features

  • Loading states during data fetching and mutations
  • Success/error message notifications (Ant Design message component)
  • Responsive table layouts that adapt to screen size
  • Professional UI using Ant Design component library
  • Form validation with helpful error messages
  • Disabled states during operations to prevent duplicate submissions
  • Modal dialogs for bug creation form

6. Application Architecture

Page Structure

File/Directory Description

api/bugs/[id]/status/route.jsPUT - update status
api/users/route.jsGET users by role
login/page.jsLogin page

Component Hierarchy

Dashboard Page (page.js):

  • Reads current user from AuthContext
  • Determines user role (admin/developer/tester)
  • Conditionally renders appropriate dashboard component

Admin Dashboard:

  • Fetches all bugs and all developers
  • Displays complete bug list in table
  • Provides assignment dropdown for each bug
  • Shows current assignee

Developer Dashboard:

  • Fetches only bugs assigned to current user
  • Displays filtered bug list
  • Provides status update dropdown
  • Shows reporter information

Tester Dashboard:

  • Displays all bugs
  • "New Bug" button opens modal form
  • Form includes title, description, priority, severity
  • Assignment capability for each bug

Data Flow

Action Flow
User Login User logs in → AuthContext stores user data (id, username, roles) → Dashboard renders based on role
Bug Creation Tester fills form → POST to /api/bugs → Backend creates record → Success message → Dashboard refreshes
Bug Assignment Select developer → PUT to /api/bugs/[id]/assign with developerId → Backend updates → Success → Refresh
Status Update Developer selects status → PUT to /api/bugs/[id]/status → Backend updates → Success → Refresh

Database Schema (Conceptual)

Users Collection:
{
  id: string,
  username: string,
  displayName: string,
  roles: string ("admin" | "developer" | "tester"),
  email: string,
  createdAt: timestamp
}

Bugs Collection:
{
  id: string,
  title: string (max 120 chars),
  description: string (max 2000 chars),
  priority: string ("low" | "medium" | "high" | "critical"),
  severity: string ("minor" | "major" | "critical"),
  status: string ("open" | "assigned" | "in_progress" | 
                  "resolved" | "rejected" | "closed"),
  creatorId: string (ref to Users),
  assigneeId: string (ref to Users, nullable),
  createdAt: timestamp,
  updatedAt: timestamp
}

Security Considerations

  • Role-based access control enforced on frontend and backend
  • API endpoints validate user permissions before operations
  • Input validation on all form submissions
  • Prevention of SQL injection through parameterized queries
  • XSS protection via React's automatic escaping
  • Session management through secure authentication
